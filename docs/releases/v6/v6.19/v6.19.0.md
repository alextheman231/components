# v6.19.0 (Minor Release)

<!-- alex-c-line-release-status-start -->
**Status**: Released
<!-- alex-c-line-release-status-end -->

<!-- alex-c-line-release-summary-start -->
This is a new minor release of the `@alextheman/components` package. It introduces new features and other backwards-compatible changes that should require little to no refactoring. Please read the description of changes below.
<!-- alex-c-line-release-summary-end -->

## Description of Changes

<!-- user-editable-section-start -->
- Add a new `createQueryBoundary` factory function.
    - This creates a compound `QueryBoundary` component containing a collection of components to interact with query data.
    - Most of these components already existed, but because they were treated separately, they did not share type information with the provider.
- Deprecate `QueryBoundary` in favour of `QueryBoundaryWrapper`
    - Although it's not a hard rule, `QueryBoundary` is the recommended name to give to the returned namespace from `createQueryBoundary`, and so just using that name as is for the wrapper can feel ambiguous.
    - As such, I have decided to rename the wrapper to `QueryBoundaryWrapper` to be more explicit.
    - The internal code for this has also been changed to use the new `createQueryBoundary` function and compound component approach.

## Migration Notes

- Add this to any component where you are currently using `QueryBoundaryProvider`, ideally inside the component function body and outside the JSX return:

```tsx
  const QueryBoundary = createQueryBoundary({
    query: { data, isLoading, error },
  });
```

- The `data`, `isLoading`, and `error` would be whatever you previously passed as those props to `QueryBoundaryProvider`.
- From there, you can pretty much just put a `.` in-between `QueryBoundary` and the component name. That is:
    - `QueryBoundaryError` -> `QueryBoundary.Error`
    - `QueryBoundaryData` -> `QueryBoundary.Data`
    - `QueryBoundaryDataMap` -> `QueryBoundary.DataMap`
    - `QueryBoundaryNullable` -> `QueryBoundary.Nullable`
    - `QueryBoundaryFallback` -> `QueryBoundary.Fallback`

## Additional Notes

- All existing separate components will remain exported in case users may still want to use them directly (e.g. to add new app-specific components to formalise query interaction per app).
- You can use this pattern with your own custom `QueryBoundary` components too. This can be done by wrapping around the original `createQueryBoundary` so you can still use our components, and then layering your changes on top. For example:

```tsx
import type {
  CreateQueryBoundaryParameters,
  DefaultQueryBoundaryComponents,
} from "@alextheman/components";

import {
    createQueryBoundary as createDefaultQueryBoundary
} from "@alextheman/components";

import QueryBoundaryDataMap from "src/components/QueryBoundaryDataMap";
import QueryBoundaryDataRowsMap from "src/components/QueryBoundaryDataRowsMap";
import QueryBoundaryProvider from "src/components/QueryBoundaryProvider";

export interface AppQueryBoundaryComponents<
  DataType,
> extends DefaultQueryBoundaryComponents<DataType> {
  Context: typeof QueryBoundaryProvider<DataType>;
  DataMap: typeof QueryBoundaryDataMap<DataType>;
  DataRowsMap: typeof QueryBoundaryDataRowsMap;
}

function createQueryBoundary<DataType>(params: CreateQueryBoundaryParameters<DataType>): AppQueryBoundaryComponents<DataType> {
  const QueryBoundary = createDefaultQueryBoundary(params);

  return {
    ...QueryBoundary,
    Context: ({ children, ...props }) => {
      return (
        <QueryBoundaryProvider
          isLoading={params.query.isLoading}
          error={params.query.error}
          data={params.query.data ?? params.query.dataCollection}
          {...props}
        >
          {children}
        </QueryBoundaryProvider>
      );
    },
    DataMap: QueryBoundaryDataMap,
    DataRowsMap: QueryBoundaryDataRowsMap,
  };
}

export default createQueryBoundary;
```

- With this approach, any new components that get added will be added to the existing set, and any components with the same name should override the default one.
- If you want to be extra safe, though, you can also omit any components from the default component set more explicitly, so you can alternatively type your app components like this:

```tsx
export interface AppQueryBoundaryComponents<
  DataType,
> extends Omit<DefaultQueryBoundaryComponents<DataType>, "Context" | "DataMap"> {
  Context: typeof QueryBoundaryProvider<DataType>;
  DataMap: typeof QueryBoundaryDataMap<DataType>;
  DataRowsMap: typeof QueryBoundaryDataRowsMap;
}
```
